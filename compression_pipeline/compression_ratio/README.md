# Compression Ratio Calculation in the Hierarchical Summarization Pipeline

## Overview

This document explains how the **compression ratio** is calculated within the hierarchical summarization architecture shown below. The objective is to provide a consistent and accurate method for measuring how effectively information is compressed as it flows through the summarization layers.

The hierarchy consists of five operational layers (L4 → L0), where information is progressively condensed:

| Level | Name       | Technique                  | Input                            | Output             |
| ----- | ---------- | -------------------------- | -------------------------------- | ------------------ |
| L4    | Sections   | Extractive (TextRank)      | Raw chunks from M1               | Section summaries  |
| L3    | Documents  | Hybrid (Extractive + BART) | L4 summaries + chunk metadata    | Document summaries |
| L2    | Topics     | Abstractive (BART)         | L3 summaries + M2 topic metadata | Topic summaries    |
| L1    | Categories | Template-Based             | L2 topic summaries               | Category summaries |
| L0    | Global     | Abstractive (BART)         | L1 categories + L2 topic labels  | Single sentence    |

---

# Compression Ratio Definition

The compression ratio is defined as:
 $CR = \frac{Input}{Output}$.

Where:

* **Input Tokens** = Total number of tokens entering a summarization layer
* **Output Tokens** = Total number of tokens produced by that layer

A higher compression ratio indicates greater information compression.

---

# Layer Flow and Token Propagation

Information flows sequentially through the hierarchy:

```text
M1 Raw Chunks and AKUs
   ↓
L4 Section Summaries
   ↓
L3 Document Summaries
   ↓
L2 Topic Summaries <- Knowldege graphs as input
   ↓
L1 Category Summaries
   ↓
L0 Global Summary
```

Each layer consumes outputs generated by the previous layer. Because of this dependency, token reuse occurs naturally across the hierarchy. The amount of tokens is reduced as compression occurs at each layer

---

# Scope of Compression Ratio Calculation

## Included Layers

The compression ratio assessment focuses on:

* **Layer 4 (L4)** as the primary evaluated output layer
* Raw input originating from **Module 1 (M1)**

## Excluded Layers

The following are excluded from the final compression ratio calculation:

### Layer 0 (L0)

L0 serves only as the final global summarization layer and is not representative of the hierarchical compression efficiency.

### Layer 2 (L2)

L2 incorporates a **knowledge graph / topic metadata** component. This metadata acts as a reference mechanism rather than introducing new informational content.

Including these reference tokens would artificially inflate the token count and distort the true compression measurement.

---

# Why Only L4 Is Used for Effective Compression Measurement

Although tokens continue through higher abstraction layers (L3 → L1), those layers derive their inputs from already summarized outputs. Including all intermediate layers would lead to repeated counting of semantically duplicated information.

Therefore, the most accurate compression measurement is obtained by comparing:

* Original raw tokens from **M1**
* Generated summary tokens at **L4**

This avoids redundancy introduced by downstream summarization stages.

---

# Effective Compression Ratio

The effective compression ratio is calculated using the total number of tokens in the original Module 1 (M1) raw chunks divided by the total number of tokens generated in the Layer 4 (L4) section summaries.

$$
\text{Effective Compression Ratio} =
\frac{\text{Tokens in M1 Raw Chunks}}
{\text{Tokens in L4 Section Summaries}}
$$

## Formula Components

- **Tokens in M1 Raw Chunks**  
  Total token count of the original input data before summarization.

- **Tokens in L4 Section Summaries**  
  Total token count after the first summarization stage (L4).

## Purpose

This calculation measures the true compression efficiency of the summarization pipeline while avoiding redundancy introduced by downstream layers and metadata references.
# Exclusion of Knowledge Graph Tokens

At Layer 2, topic metadata and knowledge graph references are injected to guide summarization quality and topic alignment.

However:

* The knowledge graph does not introduce primary source content
* It acts only as contextual reference metadata
* Counting these tokens would bias the compression ratio upward

For this reason, knowledge graph tokens are excluded from both numerator and denominator calculations.

---

# Example Calculation

Assume:

* Raw input from M1 = 120,000 tokens
* L4 section summaries = 12,000 tokens

Then:

$$
\text{Compression Ratio} = \frac{120000}{12000} = 10:1
$$

This indicates that the system compresses the original input by a factor of **10×** at the L4 stage.

---

# Final Recommendation

For consistency and accuracy, the recommended methodology is:

1. Use only **M1 raw chunk tokens** as the input baseline
2. Use only **L4 section summary tokens** as the compressed output
3. Exclude:

   * L0 global summaries
   * L2 knowledge graph / metadata references
   * Intermediate duplicated token flows

This approach provides the clearest representation of the system’s true compression efficiency while avoiding inflation caused by recursive summarization layers and metadata augmentation.

---

# Summary

The compression ratio in this hierarchical summarization pipeline is intentionally measured at the earliest meaningful summarization boundary (L4). This ensures:

* Minimal redundancy
* Accurate token accounting
* Fair measurement of actual informational compression
* Exclusion of reference-only metadata structures

The resulting metric reflects the true effectiveness of the summarization architecture rather than compounded reductions introduced by downstream abstraction layers.
