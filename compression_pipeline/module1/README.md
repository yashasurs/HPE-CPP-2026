# Module 1: Document Chunking & AKU Extraction

## Overview

Module 1 is the first stage of the AWS Knowledge Compression Pipeline. It processes raw markdown documentation and produces:
- **chunks.json**: Semantic chunks of 200-500+ tokens with metadata and hierarchical context
- **akus.json**: Atomic Knowledge Units (AKUs) extracted from each chunk in Subject-Verb-Object form

The module preprocesses documents to remove noise (TOC, page numbers, copyright), parses markdown structure, creates overlapping chunks for context preservation, and extracts domain-relevant factual statements using NLP.


## Output Files

### chunks.json

Contains structured chunks with:
- `chunk_id`: Unique identifier (e.g., "c0")
- `text`: Chunk content with 100-token overlap with next chunk
- `tokens`: Word count
- `heading_path`: Document hierarchy
- `section_id`: Source section identifier
- `source`: Source reference
- `position`: Sequential position

### akus.json

Contains extracted AKUs per chunk:
- `chunk_id`: Reference to chunk
- `akus`: Array of [Subject, Verb, Object] tuples

Example AKU:
```
["S3 bucket", "store", "objects"]
```

## How It Works

1. **Load & Preprocess**: Removes TOC lines, API versions, headers, copyright, trademarks
2. **Parse**: Converts markdown to hierarchical sections with content
3. **Chunk**: Splits sections into semantic chunks (200-500+ tokens)
4. **Overlap**: Adds last 100 tokens from previous chunk for context
5. **Extract AKUs**: Uses spaCy NLP to identify Subject-Verb-Object triples from domain text

## Configuration

Edit constants in `parser.py` to customize:

- `BAD_SUBJECTS`, `BAD_OBJECTS`, `BAD_VERBS`: Filter low-quality AKUs
- `DOMAIN_TERMS`: AWS-specific vocabulary for relevance filtering
- Chunk size thresholds in `build_chunks()` function

## Troubleshooting

**Issue**: spaCy model not found
```bash
python -m spacy download en_core_web_sm
```

**Issue**: No chunks generated
- Check that input file contains meaningful content sections
- Verify chunk size constraints match your document

**Issue**: Few or no AKUs extracted
- Content may lack domain terminology
- Review `DOMAIN_TERMS` and `BAD_*` filters for your domain

## Design decisions 

* **Separated parsing, chunking, and AKU generation**: Introduced rule-based chunking using Python’s `re` library to remove noise such as table of contents entries, page numbers, and formatting artifacts from the source documents. The cleaned and parsed output was then used for chunking and AKU generation. Chunking primarily followed a rule-based approach, while AKU extraction used spaCy as a fallback mechanism, with rule-based extraction remaining the primary strategy.

* **Initial model: `en_core_web_sm`**: The initial implementation used spaCy’s lightweight statistical model, `en_core_web_sm`, because the spaCy layer functioned only as a fallback and required fast execution. However, since the model was not trained for technical documentation, it produced low-quality and largely unusable AKUs despite aggressive filtering.

* **Transitioned to Gemini LLM for AKU generation**: Due to the limitations of the spaCy-based approach, Gemini LLM was introduced for AKU generation. This significantly improved the quality and contextual accuracy of the generated AKUs. However, the approach introduced high latency and was not scalable for extremely large datasets such as AWS documentation.

* **Switched back to spaCy using `en_core_web_trf`**: To eliminate dependency on external LLMs and improve scalability, the pipeline was migrated back to spaCy with the `en_core_web_trf` model. This transformer-based model, built on a RoBERTa backbone, handled technical language significantly better than the `sm` model. The generated AKUs showed substantial quality improvements, although inference time increased considerably.

* **Experimented with `stanza` before reverting to spaCy**: The framework was temporarily migrated to `stanza` to evaluate improvements in linguistic parsing and AKU extraction quality. While the outputs were comparable to the transformer-based spaCy model, processing time was higher with no significant quality advantage. As a result, the implementation reverted to spaCy.

* **Introduced a hybrid Gemini validation approach**: A hybrid pipeline was developed where AKUs generated by the `en_core_web_trf` model were sent to Gemini for refinement and filtering. Gemini corrected malformed AKUs and removed unusable outputs, improving overall quality. However, the approach remained unreliable due to dependence on external LLM services and associated operational constraints.

* **Enhanced parsing and chunking pipeline**: To reduce reliance on external models, the parsing pipeline was made more robust through stricter normalization, preprocessing, and document-cleaning techniques. Additionally, chunking was migrated from a purely rule-based `re` implementation to a syntax-aware approach using LangChain, improving chunk coherence and contextual segmentation.

* **Finalized with `en_core_web_trf` and advanced filtering pipeline**: The final architecture reverted to the transformer-based spaCy model combined with extensive filtering, deduplication, and scoring mechanisms. These enhancements effectively removed noisy and redundant AKUs while prioritizing high-quality outputs. The resulting system produced significantly better AKUs compared to the baseline transformer-only implementation, while remaining fully self-contained and scalable.
